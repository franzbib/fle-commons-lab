# Workflow de contribution v0.2

Ce document decrit le workflow vise par FLE Commons Lab. La v0.2 ne l'implemente pas avec un serveur: elle le prepare par types, donnees locales, pages et templates GitHub.

## 1. Deposer une ressource

Un enseignant pourra proposer une ressource en indiquant:

- niveau CECRL;
- competence principale;
- type et gabarit;
- duree;
- objectifs;
- deroule;
- support etudiant;
- guide professeur;
- corrige ou criteres;
- licence et sources.

## 2. Ameliorer une ressource

Une amelioration doit etre rattachee a une ressource existante. Elle peut corriger une erreur, clarifier une consigne, ajouter un corrige, ameliorer une progression ou rendre la ressource plus facilement adaptable.

## 3. Proposer une variante

Une variante peut concerner:

- un niveau different;
- une modalite de classe;
- une duree plus courte;
- une version numerique;
- une version sans IA;
- une version avec usage IA encadre.

## 4. Signaler un probleme

Les signalements prioritaires concernent:

- droits d'auteur incertains;
- consignes ambigues;
- erreurs linguistiques;
- difficulte technique non annoncee;
- usage IA insuffisamment encadre.

## 5. Validation future

Une future plateforme pourra distinguer brouillon, utilisable, teste, valide et archive. La validation devra rester pedagogique et collective, pas seulement technique.

## Limites de la v0.2

- pas de compte utilisateur;
- pas de formulaire persistant;
- pas de moderation;
- pas de notification;
- pas de backend;
- pas de stockage de fichier;
- pas d'appel IA.

Les issues GitHub peuvent servir temporairement a discuter du projet open source, mais l'application locale ne stocke aucune contribution.
